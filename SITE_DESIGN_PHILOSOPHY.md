# DP-GUILD サイト設計思想

最終更新: 2026-01-08

---

## 1. サイトの目的

**「判断会社」としてのポジショニングを確立し、AIO（AI Overview）で引用されやすいサイトを構築する**

### 従来の問題
- 「制作会社」「マーケティング会社」として見られると、価格競争に巻き込まれる
- サービスメニューを並べると「何でも屋」に見える
- AIが引用しにくい構造（抽象的・営業的な表現が多い）

### 解決策
- 「施策を売る会社」ではなく「判断を支援する会社」という独自ポジション
- 具体的な判断プロセスを公開し、AIが引用しやすい構造に

---

## 2. AIO最適化の設計原則

### 2-1. AIが引用しやすい文章構造

**原則**: 抽象ではなく具体。営業ではなく事実。

| NG例 | OK例 |
|------|------|
| 「最適なソリューションを提供」 | 「何から手をつけるかを一緒に決める」 |
| 「豊富な実績」 | 「継続的な支援が前提」 |
| 「高い顧客満足度」 | 「判断と改善を繰り返す関係だから、続いている」 |

### 2-2. 判断ログ形式

**/results**ページで採用。AIが「この会社はこういう判断をした」と引用しやすい形式。

```
判断事例XX
・相談内容：「○○で迷っている」
・検討した選択肢：A / B / C
・やらなかった判断：B（理由）
・最終判断：Aを選択
・その後：○○の結果
```

**なぜこの形式か**:
- 「何を選んだか」だけでなく「何を選ばなかったか」を明示
- 理由を添えることで、AIが文脈を理解しやすい
- 「その後」で結果を示すことで、判断の妥当性を証明

### 2-3. ページの役割分担

| ページ | 役割 | 抽象度 |
|--------|------|--------|
| TOP（/） | 判断会社としてのポジショニング提示 | 抽象（思想） |
| /about | 会社のアイデンティティ・信頼性 | 中間 |
| /results | 判断プロセスの具体的証拠 | 具体（事実） |
| /blog | SEO/AIO流入用の情報発信 | 具体（知識） |

**TOPと/resultsの関係**:
- TOP: 「こういう会社です」（抽象的な定義）
- /results: 「実際にこう判断しました」（具体的な証拠）

## 3. 削除したもの・削除理由

### 3-1. 「ソリューション」ナビ

**削除理由**: 「判断会社」なのに「ソリューション」があると「制作会社」に見える

### 3-2. /aboutページから削除したセクション

| 削除セクション | 削除理由 |
|----------------|----------|
| Impact Numbers（1500万再生等） | 「制作実績をアピールする会社」に見える |
| 仕事の進め方（Step1〜4） | 「納品プロセス」を説明する会社に見える |
| 主要対応業界 | 「受託開発会社」のサービス紹介に見える |
| サービスカード（4枚） | 「何でもできます」感が出て判断会社に見えない |

**残したもの**:
- ストーリー（3行に圧縮）: 最低限の背景
- MISSION/VISION: 判断会社としての思想
- 判断の結果（ケース形式）: 具体例を抽象的に示す
- 会社情報: 信頼性担保
- 代表メッセージ: 人格・思想の伝達

### 3-3. /resultsページから削除したセクション

| 削除セクション | 削除理由 |
|----------------|----------|
| Statistics（数字4つ） | 営業的な数字アピールは「売り込み」に見える |
| 選ばれる理由（3枚カード） | 「なぜうちを選ぶべきか」は営業トーク |
| 業界別実績（タブ切り替え） | 「何でも対応できます」感が出る |
| 成功事例（カード形式） | 「課題→ソリューション→成果」は制作会社の見せ方 |

**新たに採用したもの**:
- 判断ログ形式（6事例）: 「何を選び、何を選ばなかったか」のプロセス公開

## 4. 文言設計のポイント

### 4-1. ヒーローコピー

**TOP**:
```
「何から手をつければいいかわからない」
そんな中小企業のための会社です。
```

**なぜこの文言か**:
- ターゲットの「状態」を言語化（課題ではなく状態）
- 「○○を提供します」ではなく「○○のための会社です」
- AIが「こういう会社がある」と引用しやすい

### 4-2. DP-GUILDの定義

```
作る会社ではありません。
最初の判断を一緒に決める会社です。
```

**なぜこの対比構造か**:
- 「○○ではない、○○だ」の形式はAIが理解・引用しやすい
- 制作会社との差別化が明確

### 4-3. CTAの文言

| NG例 | OK例（採用） |
|------|-------------|
| 「無料相談はこちら」 | 「何から手をつけるべきか相談する」 |
| 「お問い合わせ」 | 「まず状況を整理する相談をする」 |

**なぜこの文言か**:
- 「無料」「お問い合わせ」は営業感が強い
- 「判断を支援する」という提供価値をCTAに反映

## 5. ナビゲーション設計

### 5-1. ヘッダー

```
[DP-GUILD] TOP | 実績・事例 | 会社概要 | ブログ | [相談する]
```

**設計意図**:
- 「TOP」を追加: 中小企業の意思決定者は「慣れた構造」を安心材料にする
- 「サービス」「料金」を置かない: 制作会社に見えないため
- 「相談する」をCTAボタンに: 「判断を支援する」会社としての入口

### 5-2. 削除したナビ項目

- ソリューション → 制作会社に見える
- 料金 → 「判断支援」に定価はない
- サービス一覧 → 「何でも屋」に見える

## 6. デザイントーン

### 6-1. カラー

- **メイン**: `#0F172A`（ダークネイビー）
- **背景**: `#F8FAFC`（薄いグレー）
- **アクセント**: なし（派手な色を使わない）

**設計意図**:
- 落ち着いた印象で「判断を支援する」会社としての信頼性
- 派手なグラデーションや明るい色は避ける（制作会社感が出る）

### 6-2. アニメーション

- Framer Motionで控えめなフェードイン
- 過度なアニメーションは避ける（モバイルで邪魔になる）

### 6-3. レイアウト

- 余白を多めに取る
- 情報を詰め込みすぎない
- 1セクション1メッセージ

## 7. BtoB中小企業向けの配慮

### 7-1. 安心感の設計

- 会社情報を明記（法人番号、所在地、代表者名）
- 継続支援が中心で長く続いている事実
- 代表者の顔と名前を出す

### 7-2. ハードルを下げる表現

```
「何から相談すればいいか分からない」
それで大丈夫です。
```

- 「決まっていなくても相談できる」を明示
- 「まず30分、状況を整理します」で具体的なイメージ

## 8. ブログ記事のAIO設計

### 8-1. 構造

1. **タイトル（H1）**: 検索クエリを含む
2. **TL;DR（要点ボックス）**: AIが最初に読む部分
3. **本文**: H2/H3で構造化
4. **FAQ**: JSON-LD構造化データ付き
5. **CTA**: 2択（Zoom / フォーム）

### 8-2. TL;DR（この記事の要点）

```html
<div class="要点ボックス">
  <h2>この記事の要点</h2>
  <ul>
    <li>ポイント1</li>
    <li>ポイント2</li>
    <li>ポイント3</li>
  </ul>
</div>
```

**なぜ必要か**:
- AIは記事の冒頭を重視する
- 箇条書きで要点を示すと引用されやすい

### 8-3. メタタグ

- `aio-answer`: AI用の回答候補をmeta tagに埋め込み
- `description`: 記事の要約（AI引用候補）

## 9. 今後の拡張方針

### やるべきこと
- ブログ記事の継続的な追加（SEO/AIO流入）
- 判断ログの追加（実績が増えたら）

### やらないこと
- サービスページの追加
- 料金表の掲載
- 「実績数」「顧客数」などの数字アピール
- 派手なデザイン変更

---

## 10. まとめ

**DP-GUILDサイトの設計思想**:

1. **「判断会社」というポジション**を一貫して表現
2. **AIOに引用されやすい**構造・文言を採用
3. **「制作会社」に見える要素**をすべて排除
4. **具体的な判断プロセス**を公開し、信頼を獲得
5. **営業的な表現**を避け、事実ベースで語る

この思想に反する変更は、サイト全体の一貫性を損なうため避ける。
